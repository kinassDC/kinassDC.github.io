2.概述

开发中需要用到git的场景：代码备份以防止代码丢失。代码还原，当项目中需要添加新功能为了防止越改越乱使用git可以将代码还原。协同开发，大项目一般需要多人开发，使用git可以方便多个人维护项目，当代码出现问题git可以追溯出问题的代码是谁修改。

2.1版本控制的方式

  1. 集中式版本控制工具
    集中式版本控制工具，版本库是集中存放在服务器当中，比较著名的是SVN和CVS。
  2. 分布式版本控制工具
    分布式版本控制器没有“中央服务器”，每个开发人员都有本地的本地的版本库，当然也存在远程的共享版本库，但是为了防止出现远程的共享版本库出现问题，可以将开发人员的本地版本库发送给远程的共享版本库，当开发人员修改代码之后也可以将修改后的版本库上传到共享版本库，这样就可以实现多人协同开发。比如最著名最主流的Git

3.git的工作流程

命令如下:

  1. clone(克隆):从远程仓库中克隆代码到本地仓库。
  2. checkout(检出):从本地仓库中检出一个仓库分支然后进行修订。
  3. add(添加):在提交前先将代码提交到暂存区。
  4. commit(提交):提交到本地仓库。本地仓库中保存修改的各个历史版本。
  5. fetch(抓取):从远程库，抓取到本地仓库，不进行任何的合并动作，一般操作比较少。
  6. pull (拉取): 从远程库拉到本地库，自动进行合并(merge)，然后放到到工作区，相当于fetch+merge。
  7. push(推送):修改完成后，需要和团队成员共享代码时，将代码推送到远程仓库。

5.分支

几乎所有的版本系统都支持分支。使用分支意味着可以把工作从主线上分离来进行重大的BUG修改，开发新的功能，以免影响开发主线。在多人协作开发过程中通常是一个人负责一个分支当开发完毕之后合并到主线，这样可以避免主线开发进度过于缓慢，每个分支互不影响。

5.6解决冲突

当两个分支共同更改了同一个文件分支合并的时候就会有冲突，当产生冲突的时候需要手动更改文件来选择保留哪一个分支结果。

5.7分支在开发过程中的使用流程

在一般的开发过程中有以下规范：

  • master分支（稳定分支）
    master分支一般是作为生产分支，中小规模的项目作为线上运行的应用对应的分支，是运行在服务器的机房上面，也是大多数项目上线的分支
  • develop分支（稳定分支）
    develop分支作为开发分支，一般是多数开发者所提交代码的分支。
  • feature分支
    feature分支作为开发者开发功能的分支，当功能开发完毕了需要将分支合并到develop分支，开发完毕之后该分支也就弃用了。
  • bug分支
    当上线的项目出现BUG了需要将master开一个bug分支BUG开发完毕之后合并到develop分支。

  • 生成ssh公钥
  • 获取ssh公钥 然后将输出的一段字符复制到gitee平台公钥管理入口
  • 验证配置是否成功

6.3推送本地仓库到远程仓库

  • 连接本地仓库与远程仓库
  • 查看本地仓库连接了哪些远程仓库
  • 推送本地仓库到远程仓库 参数：
    -f 本地代码如果与远程仓库有冲突则强制上传
    –set-upstream 推送到远端的同时并且建立起和远端分支的关联关系
    当配置好远程推送端对端之后可以使用git push来直接推送
    可以使用git branch -vv指令来查看这种对应关系
